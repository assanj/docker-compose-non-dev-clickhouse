НСТРУКЦИЯ ПО СИНХРОНИЗАЦИИ GIT: РЕШЕНИЕ КОНФЛИКТОВ И ПРИНУДИТЕЛЬНЫЙ PUSH
=======================================================================

СИТУАЦИЯ: Вы изменили файлы локально, но на GitHub тоже есть изменения,
и Git сообщает о конфликтах при попытке push.

ВАРИАНТ 1: АККУРАТНОЕ ОБЪЕДИНЕНИЕ (РЕКОМЕНДУЕТСЯ ДЛЯ КОМАНДЫ)
------------------------------------------------------------

ШАГИ:
1. Подтянуть изменения с GitHub:
   git pull --rebase origin main

2. Если КОНФЛИКТ (как у нас было):
   - Открыть конфликтный файл: nano имя_файла
   - Удалить маркеры конфликта (<<<<<<<, =======, >>>>>>>)
   - Оставить нужную версию кода
   - Сохранить файл

3. Продолжить объединение:
   git add имя_файла
   git rebase --continue

4. Запушить результат:
   git push origin main

ПЛЮСЫ:
- Сохраняет всю историю изменений
- Учитывает работу других разработчиков
- Профессиональный подход

МИНУСЫ:
- Требует ручного разрешения конфликтов
- Занимает больше времени

ВАРИАНТ 2: ПРИНУДИТЕЛЬНЫЙ PUSH (ДЛЯ ЭКСТРЕННЫХ СЛУЧАЕЙ)
-----------------------------------------------------

ШАГИ:
1. Принудительно заменить удаленную версию:
   git push -f origin main

2. Или более безопасный вариант:
   git push --force-with-lease origin main

ПЛЮСЫ:
- Быстро и просто
- Полный контроль над версией
- Идеально когда ваша версия - единственно правильная

МИНУСЫ:
- Стирает все изменения на GitHub
- Может вызвать проблемы у других разработчиков
- Потеря истории изменений

ВАРИАНТ 3: ПОЛНЫЙ СБРОС (КРАЙНИЙ СЛУЧАЙ)
---------------------------------------

ШАГИ:
1. Удалить локальный репозиторий:
   cd ..
   rm -rf superset-prod

2. Клонировать заново:
   git clone https://github.com/assanj/docker-compose-non-dev-clickhouse.git

3. Перенести нужные файлы и закоммитить:
   git add .
   git commit -m "New version"
   git push origin main

ПЛЮСЫ:
- Полностью чистая история
- Гарантированно работает

МИНУСЫ:
- Теряет локальные настройки
- Долгий процесс

ЧТО МЫ СДЕЛАЛИ СЕЙЧАС:
---------------------
Мы использовали Вариант 2 (принудительный пуш), потому что:
- Это наш личный проект
- Мы уверены что наша версия правильная
- Нужно было быстро решить проблему

КОГДА КАКОЙ ВАРИАНТ ВЫБИРАТЬ:
----------------------------

✅ ВАРИАНТ 1 (rebase) - когда:
   - Работаете в команде
   - Хотите сохранить историю
   - Есть время на разрешение конфликтов

✅ ВАРИАНТ 2 (force push) - когда:
   - Работаете в одиночку
   - Ваша версия - окончательная
   - Нужно быстро решить проблему

✅ ВАРИАНТ 3 (reset) - когда:
   - Все сломалось полностью
   - Конфликты слишком сложные
   - Нужно начать с чистого листа

КОМАНДЫ ДЛЯ ПРОВЕРКИ СТАТУСА:
---------------------------
- git status          - текущее состояние
- git log --oneline   - история коммитов
- git remote -v       - подключенные репозитории
- git diff            - изменения в файлах

ПРЕДУПРЕЖДЕНИЕ:
--------------
⚠️  git push -f МОЖЕТ УДАЛИТЬ ЧУЖУЮ РАБОТУ!
Используйте только когда уверены на 100%.

SSH НАСТРОЙКА ДЛЯ GITHUB

1. СОЗДАТЬ КЛЮЧ:
ssh-keygen -t ed25519 -C "ваш_email@github.com"
(на все вопросы жмем Enter)

2. ПОСМОТРЕТЬ КЛЮЧ:
cat ~/.ssh/id_ed25519.pub

3. ДОБАВИТЬ НА GITHUB:
- Зайти на https://github.com/settings/keys
- New SSH Key
- Вставить ключ из п.2
- Добавить

4. ПЕРЕКЛЮЧИТЬ РЕПОЗИТОРИЙ:
cd ~/superset-prod
git remote set-url origin git@github.com:assanj/docker-compose-non-dev-clickhouse.git

5. ПРОВЕРИТЬ:
git push origin main
(больше не спросит пароль!)
